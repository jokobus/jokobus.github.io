“Git is a distributed version control software system that is capable of managing versions of source code or data. It is often used to control source code by programmers who are developing software collaboratively.” [Wikipedia as of 2025-05-30 ]

I. Setup & Configuration

1. Install Git

(Test if installed: git --version)

Windows

winget install --id Git.Git -e --source winget

macOS

brew install git

Linux

# Debian, Ubuntu, Raspberry Pi OS
sudo apt install git

# Arch Linux
sudo pacman -S git

# Fedora
sudo dnf install git

# NixOS / Nix
nix-env -iA nixpkgs.git

2. Install GitHub CLI (gh)

Essential for PRs and repo management from the terminal.

Windows: winget install --id GitHub.cli

macOS: brew install gh

Linux: sudo apt install gh or sudo pacman -S github-cli

Authenticate:

gh auth login

3. Basic Config

Set your identity for all local repositories:

git config --global user.name "jokobus"
git config --global user.email "jokobus@example.com"

Note: Remove --global to use a different identity for just the current folder.

If using a credential manager, clear stored credentials if needed:

git credential-cache exit

4. SSH Authentication (Optional)

Recommended for secure, passwordless access.

  1. Create Key: ssh-keygen -t ed25519 -C "jokobus@example.com"
  2. Path: Press Enter for default or specify a path (e.g., ~/.ssh/id_ed25519).
  3. Add to Agent: ssh-add ~/.ssh/id_ed25519
  4. Add to GitHub: Copy the public key (id_ed25519.pub) content to GitHub Settings -> SSH and GPG keys.
  5. Test: ssh -T git@github.com
  6. Switch Remote: git remote set-url origin git@github.com:user/repo.git

II. Local Git Basics

Init

Initiate an empty git repository (in your current working directory, so cd/mkdir first). Afterwards git can track file/folder changes (add/remove/edit) in that directory.

git init

Status & Diff

  • git status: See what changes are staged/unstaged.
  • git diff: See actual changes in unstaged files.
  • git diff --staged: See changes waiting to be committed.

Stage

Add (stage) changes to the next commit.

git add <file>  # Stage a specific file
git add .       # Stage all changes in current directory
git add -A      # Stage all changes in the entire repository (including deletions)

Commit

git commit -m "Some message"
git commit -a -m "Some message" # -a commits all changes directly, skipping stage (for small changes, not recommended), or use -am

Log & Help

Log:

  • git log: Detailed history.
  • git log --oneline: Compact history.
  • git log --oneline --graph --all: Visual branch structure.
  • git log -5: Show last five commits

Tip: Use the arrow keys to navigate up/down and type :q for quit.

Help:

  • git <command> --help: Get help for a specific command.
  • git help --all: List all available commands.

Branches

Create a branch (~copy) of the main branch, edit your branch and merge it with the main branch.

git branch mybranch      # create new branch called mybranch
git branch                      # verify in which branch you are (* main)
git checkout mybranch    # therefore switch to new branch

# alternative: create and move to new branch in one line
git checkout -b mybranch

After making changes, add, commit and merge.

git add -A
git commit -m "Some message"    # Commit to mybranch
git checkout main             	# Switch back to main
git merge mybranch
git branch -d mybranch   # Delete temporary branch after successful merge

If the merge fails due to a conflict, edit the file and commit again.

III. GitHub Integration

Important: Always stay up to date (git pull origin main)! Working on outdated versions makes merging a nightmare.

1. Connect to Remote

  • Clone existing repo:
    git clone https://github.com/user/repo.git
# OR via SSH (preferred, if you have already created an SSH keypair)
git clone git@github.com:user/repo.git
  • Add remote to local repo:
    git remote add origin https://github.com/user/repo.git
git push -u origin main

2. Useful Commands

  • git fetch origin: Download latest version from GitHub (no merge).
  • git pull origin: Download and merge (Fetch + Merge).
  • git status: Show state of current repository.
  • git branch -a: Show all local and remote branches.
  • git branch -d <name>: Delete a merged branch.
  • git branch -D <name>: Force delete an unmerged branch.
  • git diff origin/main: Show differences between current branch and main.
  • git remote -v: List connected remotes (URL).
  • git remote show origin: Detailed remote information.

3. Disconnect from a GitHub Repository

git remote remove origin
git remote -v  # If nothing printed, you are disconnected.

IV. Guided Workflows

1. Simple Local Workflow

Use this for solo projects or small fixes. Tip: use branches for no matter how small the change is, as this is only a small extra overhead but an excellent practice and avoids many issues (or saves your life) when anything goes wrong.

  1. Keep updated and in sync with main (very important!):
    git pull origin main
  2. Create new branch:
    git branch mybranch
git checkout mybranch
    or as a one-liner:
    git checkout -b mybranch
  3. Make changes: In that branch, do whatever changes and commit them.
  4. Merge back to main:
    git checkout main
git pull origin main
git merge mybranch
    If you get a Merge Conflict -> resolve manually and decide whether to keep HEAD/main or edits from mybranch.
    git push origin main
  5. Post-merge:
    • Option A: Delete branch
      git branch -d mybranch
    • Option B: Reuse branch
      git checkout mybranch
git pull origin main
git merge main
      (After merging mybranch into main, merge main back into mybranch to keep it updated).
  6. Start again from step 1 for the next feature.

2. Workflow with GitHub / Pull Requests (PRs)

via GitHub Website

  1. Create, checkout and commit to a new branch.
  2. Push branch: Push your branch to GitHub.
    git push origin mybranch
  3. Create PR: Open the repo on GitHub.com and create a new PR from your branch.
  4. Review and Merge: Once reviewed, merge the PR on GitHub.
    • If changes are requested (from a coworker, repo-maintainer, etc.), keep working on that branch and commit the changes to it: the PR is updated automatically.
    • Wait until the PR is merged, before you delete the branch.
  5. Clean up: GitHub has a “Delete branch” button once the PR is closed.
  6. Start again from step 1.

via Terminal (CLI)

  1. Pull latest:

via Terminal

  1. Pull
    git pull origin main
  1. Create new Branch:
    git checkout -b mybranch
  2. Work: Make changes and commit locally.
  3. Push & PR: Push branch and create the PR from the terminal:
    git push -u origin mybranch
gh pr create --base main
    (Follow the interactive terminal prompts for Title/Body, or use --title "..." --body "..." flags).
  4. Iterate: If changes are requested, keep working locally and push. The PR updates automatically.
  5. Merge: Once approved, merge from the terminal:bash gh pr merge --merge --delete-branch
  6. Start again from step 1.

V. Utilities & Fixing Mistakes

Saving Work (Stash)

Use when you need to switch branches quickly but aren’t ready to commit.

git stash save "working on feature x" 	# Save changes
git stash list                        	# View saved work
git stash pop                         	# Bring changes back
git stash pop stash@{X}					# specifying which stash to pop (replace X)

Fixing Mistakes

  • Undo file change: git checkout -- <filename> (Discard local changes).
  • Amend: git commit --amend -m "New message" (Modify last commit message).
  • Revert: git revert HEAD (Undoes last commit safely via a new “undo” commit).
  • Reset: git reset <hash> (Moves back to a previous commit, discarding newer work). Caution: Destructive!
  • Emergency Rescue: git reflog (Shows history of all actions; find “lost” commits here).

Ignoring Files (.gitignore)

Create a .gitignore file in your repository root to skip logs or private files:

# Ignore all .log files except main.log
*.log
!main.log

# Ignore a specific directory
temp/

Note: Use .git/info/exclude for local-only ignores.

Re-evaluating .gitignore

If you added files to .gitignore that are already being tracked, you must untrack them to enforce the rules:

  1. Remove cached files:
git rm -r --cached .
  1. Re-add everything:
git add .
  1. Commit changes:
git commit -m "Untrack ignored files and apply .gitignore"

Git push branch delete to github

To delete the branch on GitHub (the remote repository), you need to push a delete command to it.

You can do this by running the following command in your terminal:

git push origin --delete mybranch

Replace mybranch with the exact name of the branch you just deleted locally.

Alternative Short Command

If you prefer a shorter shortcut, you can use a colon (:) before the branch name, which tells Git to push “nothing” into the remote branch:

git push origin :mybranch

Pro-Tip: Clean up your references

Even after deleting the branch on GitHub, your local Git history might still “remember” that the remote branch used to exist. To clean up those outdated references, run:

git fetch origin --prune

Archive Branch

Switch to or reference the branch you want to archive, and create a tag. It’s a good practice to use a naming convention like archive/.

  1. Tag the branch
git tag archive/mybranch mybranch
  1. Push the tag to GitHub
git push origin archive/mybranch

Now that the history is safely locked in a tag, you can safely delete the branch locally and remotely. Bash

  1. Delete locally
git branch -d mybranch
  1. Delete from GitHub
git push origin --delete mybranch
