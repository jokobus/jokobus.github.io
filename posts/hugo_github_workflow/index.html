Deploy Hugo on your Homserver via GitHub Actions

This guide covers how to deploy a Hugo website on your own Homserver via GitHub Actions and provides a complete, secure workflow. Your markdown files remain hidden, your compiled code remains hidden, your deployment is entirely automated, and your home server remains safely locked behind its firewall. Port 22 (for ssh) can remain closed.

Prerequisites

You need to have:

  • A GitHub account
  • A Hugo website in a private repository, pushed to GitHub
  • A Linux homeserver with Caddy Reverse Proxy running.
  • A domain (or duckdns)

If not, then this guide is not the perfect match for you (yet). However, you can get a GitHub account and set it up with this guide Git & GitHub Essentials . Another of my blogs covers How to create a blog with Hugo . Getting your own Linux homeserver is also doable, e.g. with a Raspi. Come back to this afterwards.

Summary of Workflow

[Local Desktop]     --(Push Markdown)-->    [Private Source Repo]
                                                    │
                                                    ▼
                                            (GitHub Actions Build)
                                                    |
                                                    ▼
[Home Server]       <--(Every 5 Mins)----   [Private Public Repo]
(Port 22 CLOSED)    "git pull"
  • hugo-source (Private Repo): Holds your markdown, configuration, and themes.
  • hugo-public (Private Repo): Contains only the compiled static HTML/CSS files.
  • GitHub Actions: When you push to hugo-source, the action compiles the site and automatically pushes the raw HTML into hugo-public.
  • Home Server: A lightweight system (like a script or systemd timer) pulls down the compiled code from hugo-public over HTTPS/SSH.

Step 1: Bridge the Two GitHub Repositories

Because GitHub Actions are strictly confined to their own repository by default, you need to give your hugo-source action permission to write into hugo-public.

  1. Go to your GitHub profile settings -> Developer Settings -> Personal Access Tokens -> Fine-grained tokens.
  2. Generate a new token. Name it something like hugo-compiler-token.
  3. Under Repository access, select Only select repositories and choose your hugo-public repo.
  4. Under Permissions, go to Repository permissions -> Contents and change it to Read and Write.
  5. Copy the generated token.
  6. Go to your hugo-source repository -> Settings -> Secrets and variables -> Actions -> New repository secret. Name it PUBLIC_REPO_TOKEN and paste the token.
  7. Now, add this workflow file to your hugo-source repository at .github/workflows/compile.yml:

Option A (Recommended)

name: Compile and Push to Public Repo

on:
  push:
    branches:
      - main

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout Source
        uses: actions/checkout@v4
        with:
          submodules: true

      - name: Setup Hugo
        uses: peaceiris/actions-hugo@v3
        with:
          hugo-version: 'latest'
          extended: true

      - name: Build Static Files
        # working-directory: SUBDIR #<-- Use if Hugo does not live directly in the repo, but in a subdir
        run: hugo --minify  #--baseURL "https://YOUR-DOMAIN.COM" #<-- allows to overwrite the baseURL in hugo.toml

      - name: Push to Public Repo
        uses: cpina/github-action-push-to-another-repository@main
        env:
          API_TOKEN_GITHUB: ${{ secrets.PUBLIC_REPO_TOKEN }}
        with:
          source-directory: 'public'
          destination-github-username: 'GITHUB_USERNAME'
          destination-repository-name: 'hugo-public'
          user-email: 'ME@EXAMPLE.COM'
          target-branch: 'main'

Option B (Let’s tweak)

If you followed the guide How to create a blog with Hugo , then your website/GITHUB_USERNAME/public folder might already be git initialized and contains a .git folder. This will cause several issues while running the GitHub Action, as the default cpina will try to copy your .git folder (and so the git history) from that repo to hugo-public as well. To avoid it, we change two things:

  • rsync -av --exclude='.git': Instead of a blind copy (cp), this uses rsync with an explicit exclusion rule. It takes your compiled Hugo site from hugo-source/GITHUB_USERNAME/public but completely ignores its underlying .git folder, preventing the File exists crash.

  • ! -name 'README.md': Before copying the new files over, the script cleans out the old HTML files from the target repository so deleted pages don’t pile up. The find … ! -name 'README.md' command explicitly tells the runner: “Delete the old stuff, but do not touch the README.md or the .git folder.”

Finally, the .yaml file looks as following:

name: Compile Hugo website Repo and Push to jakob.harthaus.net Repo

on:
  push:
    branches:
      - main

jobs:
  # --- Job 1 --- #
  deploy-to-homeserver:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout Source
        uses: actions/checkout@v6
        with:
          submodules: true
          fetch-depth: 0 # Fetches all history, safer for submodules

      - name: Setup Hugo
        uses: peaceiris/actions-hugo@v3
        with:
          hugo-version: 'latest'
          extended: true

      - name: Build Static Files
        working-directory: GITHUB_USERNAME
        run: hugo --minify --baseURL "https://YOUR-DOMAIN.COM"

      - name: Deploy to Target Repo
        env:
          # Using your existing secret
          API_TOKEN_GITHUB: ${{ secrets.PUBLIC_REPO_TOKEN }}
        run: |
          # 1. Configure Git
          git config --global user.name "github-actions[bot]"
          git config --global user.email "ME@EXAMPLE.COM"
          
          # 2. Clone the target repository into a temporary deployment folder
          git clone https://x-access-token:${API_TOKEN_GITHUB}@github.com/jokobus/jakob.harthaus.net.git deploy_dir
          
          # 3. Delete old website files in the target repo root, but KEEP .git and README.md
          find deploy_dir -maxdepth 1 ! -name 'deploy_dir' ! -name '.git' ! -name 'README.md' -exec rm -rf {} +
          
          # 4. Copy the newly built Hugo files into the target repo, EXCLUDING the source's .git folder
          rsync -av --exclude='.git' GITHUB_USERNAME/public/ deploy_dir/
          
          # 5. Commit and push the changes
          cd deploy_dir
          git add -A
          
          # Only commit and push if there are actual changes to prevent workflow errors
          if ! git diff-index --quiet HEAD; then
            git commit -m "Deploy website - Animated by GitHub Actions (${{ github.sha }})"
            git push origin main
          else
            echo "No changes to deploy."
          fi
  1. Commit .gitignore/workflows/compile.yml to hugo-source and push to GitHub. Now, it should compile and finish within ~1 Minute. The public code will automatically appear in hugo-public.

Troubleshoot: If you get fatal: Remote branch main not found in upstream origin on hugo-source GitHub -> Actions: This means hugo-public has no main branch. Fix: Create directly on the hugo-public GitHub website a Readme.md, write some text and commit it. This will automatically create a main branch. Then: Manually trigger the GitHub Action again in hugo-source.

Tipp: If you want to deploy the same site to a GitHub Page YOUR-GITHUB-USERNAME.github.io as well, you can do so by 1) Create a new repo called YOUR-GITHUB-USERNAME.github.io with public access right. 2) Follow the instructions as described above in Step 1 to add a new Fine-grained token and add another Actions Secret to hugo-source on GitHub. 3) Duplicate everything that is under # --- Job 1 --- # in the .yaml file. 4) Replace the job name deploy-to-homeserver by deploy-to-github-pages and replace all other placeholders accordingly (e.g. YOUR-DOMAIN.COM will be YOUR-GITHUB-USERNAME.github.io). If you followed my guide How to create a blog with Hugo , where we added the public folder as a submodule, make sure to 1) git rm --cached 2)YOUR-GIT-USERNAME/public, 2) Remove the public folder and 3) Remove all lines in .gitignores that belong to that submodule. Afterwards, all you need to do is to commit to hugo-source repo and push it to its main. GitHub Actions will take care of the compilation and pushing to YOUR-GITHUB-USERNAME repo.

Step 2: Set Up Read-Only Access for Your Server

To pull down the files securely from your private hugo-public repository without using your main GitHub password, you will use a Deploy Key. This acts as a single-purpose, read-only identity for your home server.

On your home server:

  1. Generate an SSH key:

    ssh-keygen -t ed25519 -f ~/.ssh/id_github_blog -C "homeserver-blog-puller"

    Press enter twice to leave the passphrase blank so it can run automatically.

    Keypair Excursion:

    • id_github_blog (The Private Key): This is the file without an extension. It is your server’s ultimate secret. You must never share this file, upload it to the internet, or let anyone else see it. Your server uses this secret file to prove its identity when it talks to GitHub.
    • id_github_blog.pub (The Public Key): This is the file ending in .pub. This is the “lock” that matches your private key card. It is completely safe to share. In the next steps you will take the text inside this file and paste it into GitHub as a Deploy Key.

  2. Copy the output of the public key: cat ~/.ssh/id_github_blog.pub

  3. Go to your hugo-public repository on GitHub -> Settings -> Deploy Keys -> Add deploy key.

  4. Paste the public key, name it hugo_public_to_homeserver, and leave Allow write access unchecked.

  5. Next, configure your server’s SSH client to use this specific key for GitHub by editing or creating ~/.ssh/config:

    Note: If you don’t have a ~/.ssh/ directory and ~/.ssh/config/ file yet, create them with mkdir -p ~/.ssh, chmod 700 ~/.ssh. If you have both: continue.

    Host github-blog
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_github_blog
    IdentitiesOnly yes

  6. Adjust file permissions (safeguard):

    chmod 600 ~/.ssh/config ~/.ssh/id_github_blog

  7. Test it by cloning the repo onto your home server into your Caddy folder (e.g., /var/www/hugo-public):

    sudo mkdir -p /var/www/hugo-public
sudo chown -R $USER:$USER /var/www/hugo-public

    git clone git@github-blog:YOUR_GITHUB_USERNAME/hugo-public.git /var/www/hugo-public

    Verify that the html/css/javascript is there:

    ls -l /var/www/hugo-public

Step 3: Automate the Pull

Since we don’t want a setup where the server is pinged by GitHub (as we would need to open a Port for it), it needs to check for updates on its own.

Option A: The Simple Webhook (Recommended if Caddy is Public)

Caddy can actually trigger local commands when it receives a specific URL request. You can configure a secret path in Caddy that runs git pull whenever it’s visited, then use GitHub Webhooks to ping it.

Note: While this does not open port 22, it does use your existing open HTTP/HTTPS ports (80/443) which Caddy already uses to serve your site. Therefore, I recommend to continue with Option B

Option B: The Pure Pull Method (Cron Job)

If you want zero outside dependencies triggering your server, set up a cron job to check for updates every few minutes. It uses almost zero overhead because git pull only fetches data if something actually changed.

Step 3.B.1: Open the Cron Configuration Tool

Log into your Server terminal (e.g. via ssh) and run the following command to edit your user’s automated tasks schedule:

crontab -e

Note: If it is the first time you are running this command, Linux will ask you to choose a text editor. Type the number next to nano (usually 1) and hit Enter.

Step 3.B.2: Add the Automation Line

Scroll all the way down to the very bottom of the file using your arrow keys. On a new line at the end, paste this exact configuration:

*/5 * * * * cd /var/www/hugo-public && git pull origin main >/dev/null 2>&1
  • */5 * * * *: The timing interval. It means “run this command every 5 minutes, every hour, every day.”
  • cd /var/www/hugo-public: Changes your Pi’s working directory to your website folder so Git knows which repository to update.
  • && git pull origin main: If the directory change succeeds, it runs git pull using your Deploy Key to fetch new website files.
  • >/dev/null 2>&1: This silently discards normal output and errors, preventing your Raspberry Pi from generating unnecessary system log emails every 5 minutes.

Step 3.B.3: Save and Exit

  1. Press Ctrl + O (Write Out) and hit Enter to save the file.
  2. Press Ctrl + X to exit the editor.

You should see a message in your terminal that says: crontab: installing new crontab. This means your background scheduler has accepted the task.

Step 3.B.4: Test the Pull Loop

Your secure pipeline is now officially complete. The workflow functions like an active loop:

To make sure everything is working perfectly right now:

  1. Make a small edit to a markdown file in your local source project and git push it to your hugo-source repo.
  2. Verify that your GitHub Action runs successfully and updates your hugo-public repo.
  3. Wait 5 minutes for the cron job to trigger on your Pi.
  4. Run this command on your Raspberry Pi to check if it fetched the update on its own:
cd /var/www/hugo-public && git log -1

If you see your latest commit message and timestamp from GitHub, your automation is working flawlessly! Now you can point your Caddy web server directly to /var/www/hugo-public.

Step 4: Configure Caddy

Step 4.1: Configure the Caddyfile

Caddy stores its configuration in a file called Caddyfile. Open it using nano with administrative privileges:

sudo nano /etc/caddy/Caddyfile

or

sudo nano /etc/hs_docker/Caddyfile

depending on where you house the Caddyfile.

Add your website block.

your-website.com {
    # Point Caddy to your automatically updated Hugo directory
    root * /var/www/hugo-public

    # Enable standard file serving for static HTML/CSS/JS
    file_server

    # Optional: Enable Gzip/Zstd compression for faster page loads
    encode zstd gzip
}

(Make sure to replace your-website.com with the actual domain name you used for in hugo.toml or the Hugo –baseURL override).

Save and exit (Ctrl + O, Enter, then Ctrl + X).

Step 4.2: Grant Caddy Permission to Read the Directory

Linux security requires that the Caddy background service has permission to read the files inside /var/www/hugo-public.

By default, Caddy runs under a system user named caddy. You can grant it access without breaking your user ownership by adding the caddy user to your personal user group, or by making the folder globally readable:

# Ensure directories are readable and executable, and files are readable
chmod -R 755 /var/www/hugo-public

Step 4: Reload Caddy

If Caddy was registered as systemctl service:

sudo systemctl reload caddy

If Caddy was installed via binaries:

caddy reload --config /etc/caddy/Caddyfile

If Caddy was installed via docker:

# If using standard Docker:
docker exec -w /etc/caddy caddy caddy reload

# If using Docker Compose (run this inside your docker-compose.yml folder):
docker compose exec caddy caddy reload

Congratulations!

Your modern, ultra-secure hosting stack is fully operational. Let’s look at what you’ve built:

  • Port 22 is completely closed to the outside world, rendering your Raspberry Pi invisible to SSH brute-force bots.
  • Your raw markdown source files and content drafts stay hidden away in your private source repo.
  • GitHub Actions handles the heavy lifting of compiling your Hugo site.
  • Your Pi proactively pulls updates via an encrypted SSH Deploy Key every 5 minutes.
  • Caddy serves the static assets with blazing speed and handles your SSL renewals natively.

Open up a browser, head to your domain, and enjoy your fully automated, production-grade website!

Whenever you want to update your blog, just create or edit it on your PC, revise the blog locally by looking at localhost:1313 in your browser (while running hugo server in your terminal). Once you’re done, commit and push to hugo-source. GitHub Actions and the Workflow we created will take care of the rest.
